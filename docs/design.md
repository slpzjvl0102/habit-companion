## Divergence
- **True intent / 미션 (나중에 드러남, 가장 중요):** "나는 의지박약이야"라고 믿는 사람이, *변화는 의지가 아니라 환경에서 온다* 는 걸 체감하게 하는 것. = 아주 작은 습관의 힘(Atomic Habits)의 핵심 명제. 아이용 앱은 이 미션의 한 vehicle이었음.
- Original idea:
  - 자녀의 숙제 현황, 식사 등 일상 상황을 보고하고 관리할 수 있는 앱
  - 맞벌이 부모를 위한 현황 보고 / 관리 앱
  - 자녀 입장에서 즐거움을 느낄 수 있도록 포인트, 상품, 보상 시스템
- Expansions/variants:
  - 엔진 = 자녀의 자발적 자기보고(체크) → 보상. 부모는 '감시자'가 아니라 '받아보고/반응하는 사람'으로 이동.
  - 루프 확정(의도): **자녀 체크 → 부모가 앱 내 확인 → 보상 지급** (부모 확인형 = 상호작용 루프).
  - 떠오른 심장 후보: '부모 확인' 단계가 단순 관리/승인을 넘어 맞벌이 부모-자녀의 **하루 한 번 접점("오늘 잘했네")** 이 될 수 있음. 관리앱이 아니라 연결앱이 될 여지.
  - 보상 = **두 겹**으로 좁혀짐:
    - (a) 인앱 양육(다마고치형) — 정서적 애착·매일 들여다보게 만드는 '즐거움 엔진'
    - (b) 실물 보상(용돈·간식) — 손에 잡히는 '결과 보상'
  - 둘은 서로 다른 일을 함: 다마고치=*왜 매일 여는가(재미)*, 용돈·간식=*왜 노력하는가(현실 대가)*. 원래 의도("자녀 즐거움")의 핵심은 다마고치 쪽일 가능성.
  - 결합 방식 확정(의도): **수행 → 부모 허락(게이트) → 다마고치에게 밥/포인트 소모 가능.**
  - **감정 엔진(아이디어의 심장):** 자녀가 다마고치에 애착·동질감·동정심을 느낌 → 돌보기 위해서라도 할 일을 하게 됨. = 거래(외적 보상) 루프가 아니라 **돌봄 관계** 루프.
  - 구조적 특징(평가는 보류): 부모의 '허락'이 *아이의 노력 ↔ 다마고치의 안녕* 사이에 끼어 게이트 역할. 좋게도/나쁘게도 작동 가능.
  - **새 차원: 추적 대상은 고정 체크리스트가 아니라 '점진적 습관 체인 빌더'** (ref: 아주 작은 습관의 힘 / Atomic Habits).
    - 아주 작게 시작(이불 정리) → 익숙해지면 연계 습관 추가(양치·세수) → 점점 체인 확장(이불→양치→책상→책 펴기→독서 한 페이지씩↑).
    - 원리: habit stacking(앵커 습관에 잇기), tiny start/2분 규칙, 점진적 확장.
  - 정합성 발견: 다마고치 보상 루프 = Atomic Habits 4법칙 '만족스럽게(즉각 보상)'에 정확히 대응 → 동기 엔진(다마고치)과 구조(습관 체인)가 같은 원리로 맞물림.
  - 추가 가능 레버(선택, 보류): identity-based 습관("나는 ~하는 사람"), never-miss-twice.
  - progression 방식 확정(의도): **앱 제공 템플릿** — 나이대별 또는 목표 습관별(ex. 아침 독서). 앱이 '목표 → 거기 닿는 작은 습관 체인'을 역설계해 깔아줌.
  - 파생: 앱 = 단순 트래커가 아니라 **'습관 설계 전문가'** 역할(어떤 작은 단계를 어떤 순서로 쌓아야 목표에 닿는지를 앎). 잠재 차별점 + '관리앱'→'코칭/연결앱' 성격 강화.
- Banked (나중에 validation/structuring에서):
  - 다마고치 '돌보지 않으면 시들/아픔' 류 압박 메커닉의 명암(즐거움 ↔ 죄책감/부담).
  - 부모 게이트가 누락될 때(부모가 깜빡/거부) 다마고치가 상하고 그 좌절이 부모-자녀 관계로 번질 가능성.
- Considerations surfaced:
  - 한 앱 안에 두 개의 시점이 공존함: 부모(보고받기/관리/안심)와 자녀(재미/보상). 중심축이 어디인지 아직 미정.
  - "보고" 주체 → **자녀 자기보고(자발적 체크)로 좁혀짐** (자동감지/제3자 입력 갈래는 일단 접음).
  - 새 갈림길: 자녀 체크 → 보상이 (a) 자동 지급 vs (b) 부모 확인 후 지급. 전자는 자녀 단독 루프, 후자는 부모-자녀 상호작용 루프. (위 '두 시점'이 여기서 다시 만남.)
- Prior art (descriptive):
  - (아직 안 다룸)

## Validation
- Load-bearing assumptions (받침대 — 무너지면 아이디어 전체가 흔들리는 가정):
  - **A. 부모가 (바쁜 맞벌이임에도) 제때 확인/허락해준다.** 보상 즉시성(Atomic Habits 4법칙)이 여기 달림. 타깃 유저(바쁜 부모)가 곧 루프의 병목 → 가장 약한 받침대. ✅ **완화됨**: '반응/보상 분리'로 즉시성은 게이트 밖에서 확보.
  - B. 아이의 다마고치 애착이 충분히 오래 간다 (노벨티 소멸 / 연령 의존성).
  - C. 자기보고가 정직하다 / 검증 가능하다 (부재중 부모는 물리적 검증 불가 → 묵인 시 게이밍, 검증 강요 시 불가능. A와 얽힘).
- Feasibility findings:
  - **A 해결 시도 (사용자 제안 2개):**
    1. 승인 모드 옵션화(즉시 지급 / 부모 확인) + 템플릿 기본값. → A를 *해결*이 아니라 '각 가족이 견딜 버전의 A를 고르게'. '확인' 모드엔 병목 잔존. ⇒ **템플릿 기본값 = 사실상 진짜 설계 결정.**
    2. 전날 수행 → 다음날 다마고치 밥/실물 정산. → 부모에게 '저녁 창'을 줘서 **병목을 구조적으로 해소.** BUT 즉각 보상(Atomic Habits 4법칙)을 깎음. **연령 의존: 어릴수록(5~7세) 지연 타격 큼.**
  - **제안된 합(구조 영역 — 사용자 확인 대기):** '반응 / 보상' 분리.
    - 즉각 반응(게이트X·공짜): 체크 순간 다마고치가 바로 반응 → 감정적 즉시성 확보.
    - 정산 보상(게이트O·지연OK): 진짜 밥/포인트/실물은 부모 확인 또는 전날-기반 다음날 정산 → 부모 통제·저녁 창 확보.
  - **반복 등장하는 핵심 변수: 타깃 나이대.** A(지연 내성)·B(애착 지속)·C(자기보고 정직)를 모두 좌우.
  - **타깃 나이대 지정(사용자): 유아 ~ 초3·4학년 (약 4~10세).** Mode C 지적:
    - 띠가 넓어 사실상 **두 사용자**. 기기/문해/자율이 갈림: 유아=자기 폰 없음·미독해·부모 동반 / 초3·4=독해·단독 앱 사용 가능(단, 게이밍 가능).
    - **사용자 습관 예시(책상→책 펴기→독서 한 페이지)가 윗쪽(읽는 나이)에 쏠림** — 4세는 해당 없음.
    - 호재: 다마고치 애착(B)은 이 구간 전체에서 통함('유치함' 리스크는 11세+).
    - 갈림길(결정 대기): ① grow-with-child(이상, 연 단위 확장 = 원 철학과 정합, 단 v1에서 양 끝 동시 충족은 둘 다 어설퍼질 위험) vs ② 웨지 먼저(한 끝부터). **권고: v1은 초1~4(7~10세) 앵커** — 자기보고-온-디바이스가 실제 작동 + 습관 예시가 거기 삶. 유아는 같은 펫으로 추후 하향 확장.
- **가장 큰 받침대(미션 정렬) — 결정 대기:** 미션의 대상('의지박약' 자기서사를 가진 청소년·어른)과 현재 앱의 대상(4~10세 아이)이 **어긋남.** 어린아이는 아직 그 자기서사가 없음. ⇒ vehicle 갈림길:
  - (가) **예방**: 아이용 유지 + 미션으로 재해석 — 어릴 때 "환경이 날 바꿨다"를 경험시켜 '의지박약' 믿음이 *애초에 안 생기게*.
  - (나) **치료**: 타깃을 상처 있는 청소년·어른 본인으로 이동 — 1인용 환경설계 습관 앱.
  - (다) **부모가 진짜 관객**: 아이 변화를 본 부모(흔히 그 의지박약 어른)가 자신에 대해 깨달음.
  - 호재: 어느 쪽이든 기존 엔진(반응/보상 분리·다마고치 돌봄·점진 습관 체인·환경 템플릿)은 거의 전이됨. 바뀌는 건 '운전석'.
  - **미션 = 개인적/실화:** 사용자 본인이 '의지박약' 믿음을 이미 극복 → 그 경험을 나누고 싶음. (founder-problem fit = 최강 토대.)
  - **사용자 제안: 한 앱, 두 모드 — 싱글(자기-도전, 의지박약 본인)/듀얼(부모-자녀, 유아용).** = 갈림길의 (가)+(나) 둘 다.
  - **Mode C 지적:** 두 '모드' ≈ 두 '제품'. 엔진은 공유하나 코어 루프가 갈라짐. 결정적: **싱글엔 '부모 확인' 게이트가 없음** → 정산 보상 매듭 주체 부재 → **받침대 C(자기보고 정직)에 백스톱 없이 전적 의존.** ⇒ 토글이 아니라 사실상 두 번 설계.
  - 시장 지형(Mode A): 싱글(어른용 귀여운 생물 자기관리 앱)은 붐빔(예: Finch). 듀얼(부모 게이트+가족 다리)은 덜 밟힌 땅이며 메커니즘 대부분 이미 설계됨.
  - **권고: v1은 한 모드만 확실히.** 미션 직격=싱글(본인 전환 경험의 재현, 아이는 그 '상처/아하'가 없음) vs 차별화·준비도=듀얼. **결정 대기.**
- Decisions:
  - **[확정] 보상 구조 = '반응 / 보상' 분리.** 즉각 반응(게이트X·공짜·감정 즉시성) + 정산 보상(게이트O·부모 확인 또는 전날기반 다음날·부모 통제/저녁 창). → 받침대 A 완화.
  - **[확정] v1 웨지 = 듀얼 모드 · 유아용 · 관리자(부모) ↔ 실행자(자녀).** 싱글은 비전에 남기고 시퀀싱(듀얼 먼저). '관리자/실행자' 추상화가 (1) 유아 기기/문해 크랙 해소(부모가 admin 운전, 아이는 현실 행동+펫 시청), (2) 추후 코치-클라이언트/교사-학생 확장 여지.
  - **[v1 강제 제약 — 구조 단계로 이월] 두 가지:**
    1. **부모-아하 필수.** 유아는 '의지박약' 자기서사 없음 → 미션('환경>의지' 체감)을 받을 사람은 관리자(부모)뿐. v1에 *부모가 "잔소리/의지 아니라 환경이 바꿨네"를 느끼는 순간* 이 없으면 미션 증발 → 평범한 칭찬스티커 앱으로 전락.
    2. **유아 펫 무처벌(굶김/죽음 금지) — 단, '슬픔'은 허용.** [사용자 정정] 굶기거나 죽이지는 않되 *경미한 슬픔* 은 둠. **슬픔 조건 = (관리자 정산 안함) AND (실행자 실행 안함).** AND라서 아이가 했는데 부모가 깜빡한 날엔 펫이 안 슬픔(아이를 부모 실수로부터 보호).
  - 부수: 받침대 C(자기보고 정직)는 유아+부모 동석 config에서 위험 급감(유아 게이밍 거의 불가).

## Structuring
- **펫 상태기계 (v1) — 2층 구조 [✅ 확정]:**
  - **층1 — 펫의 기분 ← 아이의 행동(1차, 주인=아이):** 체크 → 기뻐함(즉시). 안 함 → 시무룩(하루 경계). ⇒ 아이가 펫의 '마음'을 쥠. 부모는 기분 못 바꿈 → **스테이크 유지형**(구제 레버 질문 자동 해소).
  - **층2 — 펫의 밥/포인트/실물 ← 부모의 정산(2차, 주인=부모):** 정산 → 펫 실제 식사 + 포인트/실물. 미정산 → 밥 '대기'(기분은 이미 좋음).
  - 파생 상태: **"신났는데 아직 밥 대기"**(아이 O · 부모 확인 대기) — 아이 안 혼내고 부모에게 "밥 기다려요" 신호. 단 '굶주림' 아닌 '설레며 기다림'으로.
  - **이 2층 모델이 이전 평면 AND를 대체.** child-protection 속성은 더 좋게 유지(아이 O면 펫은 '행복', 부모 지연은 '밥 대기'일 뿐).
  - 제약 유지: 슬픔은 경미+즉시 회복(누적/악화 금지), 인스턴트 순간엔 슬픔 없음.
- **휴식일 처리 [✅ 확정]:** 부모 '오늘 쉬어요' 토글 → 그날 펫이 *잠/여행*(긍정 상태). 시무룩 평가 정지 + 습관 진행 *일시정지(끊김 없음)*. 유아한텐 '실패'가 아니라 '펫도 쉬는 날'. ⚠️ 가시적·의도적 상태로 설계해 '매일 도피처(구제 레버 뒷문)' 방지.

- **하루 코어 루프 (실행자=아이 입장) [조립]:**
  - 🌅 아침: 오늘의 고리 표시(예: 이불 정리 → 양치) · 큰 그림 버튼.
  - ✅ 아이 "했어!" 탭 → 그 순간 펫이 좋아함(게이트X·즉시).
  - ⏳ 낮: 펫 "신났는데 밥 대기"(부모 확인 대기).
  - 🌙 저녁: 부모가 정산(확인) → 펫 실제 식사 + 포인트 + (약속된) 실물.
  - 🌑 밤(롤오버): 했으면 행복/성장, 양쪽 다 안 했으면 경미 시무룩. 고리 익으면 다음 고리 얹힘. (쉬는 날=잠/여행.)
- **두 역할 [조립]:**
  - 실행자(아이): 현실 행동 + 펫 보기 + "했어!" 탭. UI=그림·큰 버튼(미독해 OK). 부모 기기에서 함께.
  - 관리자(부모): 고리 템플릿 선택, 정산(확인), 실물 보상 약속, '부모 아하' 수신.
- **'부모 아하' 순간 [✅ 확정 = A+B]:** 주간 '성장 스토리' 카드(A, 서사+명시 귀인) 안에 '그때 vs 지금' 시각(B, 성장 증거)을 결합. B=증거, A=의미 읽기. 미션(환경>의지) 수신자=부모. (1주차엔 '그때' 없음 → '여기서 출발' 베이스라인, 대비는 몇 주 후부터.) (C)는 후순위.
- **MVP 경계 [✅ 확정]:**
  - IN(최소 가설 테스트): 부모1+아이1+펫1 / 고리 템플릿 1개(이불 정리→최대 3 스택) / 아이 셀프체크+즉시 반응 / 부모 정산→식사+포인트 / 펫 2층 상태기계+휴식일 / 포인트→펫 먹이기 / 주간 스토리 카드(A+B).
  - DEFER: 실물 보상=부모 수동 약속 메모만(결제/용돈 연동 ❌) · 싱글 모드 · 템플릿 라이브러리 · 펫 커스터마이즈/멀티펫 · (C) · 코치-학생 확장.
  - 진행 트리거 [확정 기본값]: 최근 7일 중 5일 성공 → 앱이 다음 고리 제안 → 부모 승인.

## Office-Hours (수요 진단)
- 모드: Startup / pre-product.
- **수요 증거: 없음(사용자 본인 확인).** 본인 말고 이 문제로 *행동한* 부모 사례 0.
- 핵심 리스크: 현상유지(폰=무노력 즉시 평온)가 '부모 노력 0'으로 이김. 제품은 부모에게 매일 노력 요구 → 가장 시간 없는 사람에게 더 많은 일을 요구. **죄책감 ≠ 수요.**
- 관찰: 유일한 founder-problem-fit은 *어른/싱글* 버전(본인 극복). 먼저 만들기로 한 유아·듀얼엔 증거 0.
- 프레임 재정의(사용자): 앱의 1차 목적 = "환경이 사람을 이끈다" 가설 입증.
  - 진단: (1) 그 가설은 이미 정설(Nudge·Fogg·Wood·ego-depletion 재현실패). (2) 좁은 주장(이 메커니즘이 유아 습관 형성)은 미검증이나 *제품이 아니라 실험* 으로 증명. (3) '앱으로 증명'도 실사용자 필요 → 수요 회피 불가.
- **결정(사용자): 일단 앱을 만들어 실험.** (권고였던 종이 N=1 대신.) → 조건: 빌드를 *실험* 으로 설계.

## 실험 설계 (build-as-experiment)
- 가설(1문장, falsifiable): 유아가 다마고치를 돌보려는 마음 때문에, 잔소리 없이 아침 습관 하나를 스스로 하게 된다.
- 성공/실패 기준(시작 전 확정): **성공** = 3주 후 아이가 첫 습관을 스스로(부모 안 깨워도) 주 5일+ & 부모가 '환경이 했다' 체감. **실패** = 여전히 매일 시켜야 함 / 아이가 펫에 흥미 잃음.
- 실험-등급 빌드(제품-등급 ❌, MVP보다 더 자름): 앱스토어 배포·펫 아트·템플릿 시스템·회원가입 전부 제외. 핵심 루프(체크→펫 반응→부모 정산→식사→주간 카드)만 ONE 기기 ONE 아이로.
- **실제 피험자 1명 사전 확보 필수**(없으면 사용자 0 = 증명 0). ✅ **확보됨(사용자 확인).**
- **설계 규칙: 그 아이가 곧 명세.** 첫 습관은 템플릿이 아니라 *그 아이의 실제 현재 루틴* 에서, 2분 규칙으로 '실패 불가능'할 만큼 작게 (이미 거의 하는 것에 살짝 얹기).
- 종료 시 결정 사전 약속: 붙으면 키움 / 안 붙으면 메커니즘 수정 or 중단.
- **스택 = Flutter (사용자 선택).** 적합성: 펫 무드 애니메이션이 정서적 핵심 → Flutter 애니메이션 강점과 맞음. ⚠️ 주의: 셋업·반복 비용이 2~4주 실험 타임라인 먹지 않게 범위를 실험-등급으로 고정. 로컬 저장만(백엔드 0).
